# 分支发布工作流（integration 主线）

## 1. 分支角色
1. `main`：仅保留可发布资产，禁止直接推送。
2. `integration`：发布前集成主线，所有功能分支先合入此分支。
3. `feature-async-incubator`：`lib/async` 长期开发分支。
4. `feature-services-incubator`：`lib/services` 长期开发分支。
5. `release/x.y`：从 `integration` 切出，执行发布清理、验证后合并到 `main`。
6. `hotfix/*`：从 `main` 切出，修复后回灌 `integration`。

## 2. 合入路径约束
1. 常规需求：`feature/* -> integration`。
2. 发布路径：`integration -> release/x.y -> main`。
3. 紧急修复：`main -> hotfix/* -> main`，随后 `cherry-pick` 回灌 `integration`。

## 3. `main` 发布门禁
1. `main` 不允许包含下列过程文档目录：
   - `lib/async/doc/**`
   - `lib/services/doc/**`
   - `lib/services/src/**/doc/**`
2. `main` 不允许存在对上述目录的残留引用。
3. 代码风格与命名检查暂采用本地执行：发布前执行 `xmake format --dry-run --error` 与 `xmake check clang.tidy`（自动发现目录配置）。
4. 当前阶段暂不启用 CI 强制门禁，仍保留人工复核，重点确认过程文档清理与引用完整性。

### 3.1 clang-tidy 官方自动修复命令
1. `xmake check clang.tidy --fix`
2. `xmake check clang.tidy --fix_errors`
3. `xmake check clang.tidy --fix_notes`
4. 官方参考：<https://xmake.io/zh/guide/extensions/builtin-plugins.html#%E6%A3%80%E6%B5%8B%E5%B7%A5%E7%A8%8B%E4%BB%A3%E7%A0%81-clang-tidy>
5. 职责边界：根 `.clang-tidy` 负责默认规则；`platform/.clang-tidy` 使用 `Checks: '-*,clang-diagnostic-*,llvm-twine-local'` 且 `InheritParentConfig: false`，使平台目录仅保留编译诊断（`llvm-twine-local` 仅用于规避 `no checks enabled`）；源文件扫描范围可由 `xmake check clang.tidy --files=...` 在本地按需缩小。

## 4. 发布清理步骤
1. 在 `release/x.y` 人工清理以下过程文档目录：
   - `lib/async/doc/**`
   - `lib/services/doc/**`
   - `lib/services/src/**/doc/**`
2. 人工搜索并修复对上述目录的残留引用（重点检查 `README.md`、`document/**`、`lib/**/README.md`）。
3. 完成构建和回归验证后，方可发起 `release/x.y -> main` PR。

## 5. 协作建议（Git 高级工作流）
1. 长分支迭代前先 `rebase integration`，保持提交线性。
2. 清理提交历史时使用 `rebase -i`，将修补提交折叠成原子变更。
3. 改写历史后统一使用 `push --force-with-lease`，禁止裸 `--force`。
4. 发生误操作时使用 `reflog` 恢复。

## 6. 命名与兼容说明
1. 仓库当前已存在分支名 `feature`，不能再创建 `feature/xxx` 命名空间分支。
2. 因此采用 `feature-async-incubator`、`feature-services-incubator` 作为等价长期 feature 分支命名。
