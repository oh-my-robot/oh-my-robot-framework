# 电机驱动模型架构设计（OM 落地版）

## 0) 适用范围与目标

- 总线：CAN、RS485（基于串口设备）
- 供应商：DJI、达妙、宇树、本末
- 目标：通信与安全为主 + 内置控制能力
- 非目标：EtherCAT 等保留能力编号，但不作为近期实施范围

## 2) OM 落地约束与接口约定

- Bus 与设备统一使用 `Device_t` 与 `device_*` 接口（`device.h`）
- CAN：通过 `device_ctrl(CAN_CMD_FILTER_ALLOC, ...)` 申请过滤器并注册 `rx_callback`；收发使用 `device_read/device_write`
- CAN 时间戳来源：`CanMsgDsc_s.timeStamp`（`pal_can_dev.h`）
- 串口/RS485：基于串口设备接口读写与回调；当前头文件未提供 RS485 专用控制接口，若需方向控制应在 BSP 或协议层补充
- 多 bus 实例：通过 `device_register/device_find` 管理；以设备名区分
- 内存与实时：优先静态分配与 O(1) 路由；DJI 驱动已采用静态池与映射表

### 术语定义与推荐设置

- **bus 定义**：指物理通信链路/设备实例（如 CAN1、CAN2、某条 RS485 链路），是带宽与时序资源的共享边界。
- **per-bus flush 频率**：总线级“统一发送节拍/预算周期”，每次 flush 负责打包并下发本周期内可发送的命令。
- **per-motor tx_hz**：电机级“期望发送节拍”，用于在 flush 时决定是否更新该电机的命令。

**控制/发送/反馈频率关系**
- **控制频率**：控制算法/参考更新频率（生成 `ref/cmd` 的节拍）。
- **发送频率**：总线实际发帧频率（`tx_hz` 与 `bus_flush_hz` 的组合结果）。
- **反馈频率**：电机回报的实际节拍（协议/驱动/总线共同决定）。
- **闭环上限**：实际闭环更新效果通常受 `min(控制频率, 发送频率, 反馈频率)` 约束。

**示例**
- 控制 1kHz、发送 500Hz、反馈 1kHz：实际指令更新受 500Hz 限制（每 2ms 有效更新一次）。
- 控制 200Hz、发送 1kHz、反馈 1kHz：总线多次发送相同命令，实际指令更新受 200Hz 限制。

**同模块电机通常在同一总线时的推荐设置**
- **统一频率模式**：所有电机 `tx_hz == bus_flush_hz`。  
  适用：电机数量少、带宽足、优先简单可控。
- **分级频率模式**：`bus_flush_hz` 高于或等于关键电机的 `tx_hz`，非关键电机设为低频。  
  适用：带宽紧张、不同关节响应需求差异大，需要做带宽治理。

## 3) 电机框架完整能力模型（Capability Model）

> 说明：能力编号格式 `MF-类别-序号`。
> 你后续所有需求、任务、测试用例都可以引用这些编号，避免“口头描述漂移”。

### A. 总线与传输层（Bus/Transport）

- **MF-BUS-01 总线抽象接口**：统一 CAN/RS485 的 send/recv API（基于 `Device_t`），接口稳定。
- **MF-BUS-02 RX/TX 队列与上下文隔离**：ISR 只入队，解析在任务/主循环完成；可配置队列长度与溢出策略。
- **MF-BUS-03 帧时间戳**：为 RX 帧附加单调时钟时间戳（CAN 可直接使用 `timeStamp`）；用于超时/抖动统计与控制诊断。
- **MF-BUS-04 多 bus 管理**：支持多个总线实例并行运行；bus 上挂载多个 motor。
- **MF-BUS-05 带宽/节拍治理**：支持 per-bus flush 周期、发送预算、拥塞保护（丢弃/降频/延后）。
- **MF-BUS-06 group frame 聚合**：支持协议允许的合帧（pack/flush），具备 slot/索引管理与策略（HOLD_LAST/OMIT/SAFE）。
- **MF-BUS-07 错误与链路健康**：bus-off/丢帧/队列溢出统计与事件上报。

### B. 协议驱动层（Protocol Driver Plugin）

- **MF-PROTO-01 驱动插件接口**：统一 `init/enable/disable/encode_cmd/decode_fb` 等 vtable。
- **MF-PROTO-02 多型号适配**：DJI/达妙/宇树/本末以“新增一个 driver 文件”方式扩展，不侵入 core。
- **MF-PROTO-03 ID 映射与寻址规则**：支持按 CAN ID/设备地址/协议 ID 定位 motor。
- **MF-PROTO-04 单位换算与标度**：协议内部原始量映射为框架统一单位（rad、rad/s、N·m、A、°C）。
- **MF-PROTO-05 方向/减速比/零点补偿**：将关节侧量统一呈现（dir/gear/offset）。
- **MF-PROTO-06 厂商故障码映射**：将 vendor code 映射为统一 `MotorFaultCode`。
- **MF-PROTO-07 参数读写通道（可选）**：若协议支持，提供 set/get 参数能力（模式、限幅、零点、PID 等）。
- **MF-PROTO-08 协议一致性校验**：CRC/序号/字段有效性检查（若协议支持则启用）。

### C. 电机对象模型（Motor Core Object Model）

- **MF-CORE-01 Motor 注册与生命周期**：注册、反注册（可选）、启用/禁用、初始化状态清晰可控。
- **MF-CORE-02 统一数据模型**：`Ref/Cmd/Fb` 结构 + `valid_mask` + `timestamp` 语义明确。
- **MF-CORE-03 运行状态机**：至少包含 READY/ENABLED/RUNNING/FAULT/OFFLINE（可扩展 DEGRADED/ESTOP）。
- **MF-CORE-04 配置驱动的能力声明**：注册时通过参数声明 caps/allowed modes/频率/限幅等（不做自动协商）。
- **MF-CORE-05 模式与通道管理**：支持 torque/vel/pos/MIT 等模式（至少能“声明/选择/限制”）。
- **MF-CORE-06 并发一致性**：ref/cmd/fb 在多上下文读写下保持一致（临界区或 seqlock）。

### D. 调度与频率模型（Scheduling & Rates）

- **MF-SCH-01 模块控制频率解耦**：上层模块可按自身频率写 ref/cmd，框架不强绑定控制频率。
- **MF-SCH-02 per-motor TX 频率**：每个 motor 可配置 tx_hz，框架在 flush 时做分频与策略（HOLD_LAST 等）。
- **MF-SCH-03 per-motor 反馈期望与超时**：每 motor 维护 `fb_expect_hz` 与 `fb_timeout_us`，用于健康检查。
- **MF-SCH-04 per-bus flush 调度**：每条 bus 有 flush 周期或预算机制，适配 group frame 与带宽限制。
- **MF-SCH-05 抖动与 dt 统计（可选）**：统计控制/发送/反馈的 jitter，用于诊断。

### E. 安全约束与限幅（Safety Envelope）

- **MF-SAFE-01 统一限幅器**：力矩/速度/位置软限位、输出限幅、可配置优先级与策略。
- **MF-SAFE-02 热保护（降额/跳闸）**：temp_warn 限扭；temp_trip 禁能/锁存。
- **MF-SAFE-03 命令过期策略**：上层 ref/cmd 长时间不更新时的行为（保持/缓降/禁能）。
- **MF-SAFE-04 安全启动序列**：上电/使能前的“零输出/握手/反馈稳定”策略，避免上电冲击。
- **MF-SAFE-05 急停通道（可选）**：系统级 estop 触发全电机进入安全状态。

### F. 故障与恢复（Fault Strategy）

- **MF-FLT-01 统一故障分类**：通信/电气/热/传感/控制/配置等统一 fault code。
- **MF-FLT-02 故障规则表（可配置）**：fault→action（WARN/LIMIT/DISABLE/LATCH/RECOVER）。
- **MF-FLT-03 锁存与清除条件**：手动清、自动清、回落清、重试次数等。
- **MF-FLT-04 通信质量统计**：timeout、seq loss（若有）、CRC error、bus error 等。
- **MF-FLT-05 恢复策略（可选）**：重连、退避、自动 re-enable、恢复窗口。

### G. 控制器集成（Controllers）

> 本项目需要内置控制能力：框架提供控制器插件接口与内置实现，上层可选择直用或仅透传。

- **MF-CTL-01 控制器插件接口**：ControllerOps（reset/update/set_params）。
- **MF-CTL-02 内置 PID**：带 anti-windup、微分滤波、饱和计数。
- **MF-CTL-03 内置 MIT**：kp/kd/ff → torque 命令的标准实现或透传。
- **MF-CTL-04 级联/组合控制**：position→velocity→torque，或 limiter/feedforward 链式组合。
- **MF-CTL-05 控制与发送解耦**：控制输出与 TX flush 解耦（支持模块控制频率≠tx_hz）。

### H. 轨迹与参考整形（Trajectory / Reference Shaping）

- **MF-TRJ-01 参考斜坡/限加速度**：rate limiter（低成本、强实用）。
- **MF-TRJ-02 单关节轨迹**：梯形速度/S 曲线（jerk 限制）。
- **MF-TRJ-03 多轴同步**：同模块多电机同步到达/同步启停（通常在上层模块实现）。

### I. 观测与滤波（Observer / Filtering）

- **MF-OBS-01 原始与估计分离**：fb_raw / fb_est 的模型（即使首版不实现滤波也保留结构位）。
- **MF-OBS-02 基础滤波器**：LPF/差分速度/α-β。
- **MF-OBS-03 高级观测**：卡尔曼/融合（通常更上层）。

### J. 诊断与可观测性（Diagnostics & Observability）

- **MF-DIAG-01 last snapshot**：每电机保存 last ref/cmd/fb/state/fault（低成本但关键）。
- **MF-DIAG-02 指标 metrics**：rx/tx、timeouts、saturations、temp_peak、bus_errors 等。
- **MF-DIAG-03 事件记录（轻量）**：最近 K 条事件（FAULT_SET/STATE_CHANGE/TIMEOUT…）。
- **MF-DIAG-04 Motor_Dump 统一输出**：可接串口/日志系统。
- **MF-DIAG-05 trace ring（可选）**：最近 N 次 ref/cmd/fb 全量或采样记录。
- **MF-DIAG-06 结构化日志（可选）**：二进制/CBOR/JSON 等导出格式，便于回放。

### K. 配置与参数（Config & Parameters）

- **MF-CFG-01 静态配置模型**：每 motor 配置：caps、mode、限幅、频率、超时、bus、slot 等。
- **MF-CFG-02 运行时参数更新（可选）**：在线调参（安全校验、权限、回滚）。
- **MF-CFG-03 持久化与版本（可选）**：参数存储、版本兼容、CRC 校验。

### L. 工程化：资源、可裁剪、测试（Engineering）

- **MF-ENG-01 宏开关裁剪**：DIAG/TRACE/PROTO/FEATURE 编译期裁剪，零成本关闭。
- **MF-ENG-02 静态内存策略**：避免 malloc；ring/queue 静态分配；可配置容量。
- **MF-ENG-03 性能与预算**：每 tick 最大处理帧数、flush 预算，避免单点拖垮实时性。
- **MF-ENG-04 仿真/回放（可选）**：mock bus、record/replay、故障注入。
- **MF-ENG-05 API 版本治理（可选）**：接口版本号、兼容策略。

## 5) 分阶段能力实现清单（OM 定制）

> 目标路径：**“通信可用” → “安全可用 + 内置控制” → “多协议落地” → “工程化完善”**。

### Phase 0：总线与插件骨架跑通（CAN + DJI）

**目标**：在 OM 现有 CAN 框架上跑通最小闭环，具备插件机制与统一数据模型骨架。  
实现能力：

- MF-BUS-01、MF-BUS-02、MF-BUS-03、MF-BUS-04（基于 `Device_t` + CAN）
- MF-PROTO-01（插件接口）+ 先适配现有 DJI 驱动为首个 proto
- MF-CORE-01、MF-CORE-02、MF-CORE-04、MF-CORE-06
- MF-SCH-04（per-bus flush 基础框架）
- MF-DIAG-04（最简 Motor_Dump）
**验收点**：
- 2~4 个电机注册成功，enable/disable 有效
- RX 更新 fb，TX 周期发送（允许先用 DJI 驱动验证）

---

### Phase 1：安全可用 + 内置控制（稳定上车）

**目标**：上车可稳定运行，具备最小安全网与内置控制闭环能力。  
实现能力：

- MF-SCH-02、MF-SCH-03（per-motor tx_hz + fb_timeout）
- MF-CORE-03（状态机 READY/ENABLED/RUNNING/FAULT/OFFLINE）
- MF-FLT-01、MF-FLT-02（至少 TIMEOUT → DISABLE/LATCH）
- MF-SAFE-01（基础限幅：torque/vel）
- MF-CTL-01、MF-CTL-02（控制器接口 + 内置 PID）
- MF-DIAG-01、MF-DIAG-02（last snapshot + metrics）
**验收点**：
