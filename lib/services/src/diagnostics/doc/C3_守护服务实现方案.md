# C3_守护服务实现方案

## 1. 背景与目标

基于 `C3_守护服务语义冻结.md`，本文给出 OM 守护服务的实现级方案，目标是在不破坏现有分层与接口兼容性的前提下，落地统一在线监测能力。

实现目标：

1. 支持驱动、线程、服务等实体统一注册与租约续约。
2. 实时判定在线状态并触发边沿事件。
3. 离线时执行“上报 + 可注册回调”，不强制自动恢复。
4. 通过适配层桥接现有驱动在线语义，避免直接改动既有公开 API。

## 2. 范围与非目标

### 2.1 范围

1. `services/diagnostics` 内守护控制面与注册中心实现。
2. 依赖 `async` 抽象的异步分发执行链。
3. 统一事件输出到 `diagnostics/event/log` 观测链路。
4. 面向现有驱动路径的桥接接入方案。

### 2.2 非目标

1. 不新增强制类型化实体枚举作为核心主键。
2. 不在守护核心实现业务恢复策略（重启、重建、隔离）。
3. 不在本阶段改动现有驱动对外 API 签名。
4. 不绑定具体 RTOS 队列或特定芯片 watchdog 外设。

## 3. 依赖与边界

1. 依赖边界遵循 `oh-my-robot/document/architecture/分层与依赖规范.md`。
2. 守护控制面位于 `lib/services/diagnostics`，不反向依赖 `drivers(core)`。
3. 执行面依赖 `lib/async` 抽象能力，不直接依赖具体队列实现。
4. 驱动适配层负责桥接事件，守护核心不感知驱动私有结构。
5. `systems` 负责恢复策略与业务动作编排。

## 4. 方案与实现要点

### 4.1 组件划分

1. `guard_registry`：实体注册表与租约状态中心（单一事实源）。
2. `guard_detector`：基于单调时钟执行超时扫描与状态推进。
3. `guard_dispatcher`：边沿事件异步分发与回调执行。
4. `guard_adapter`：驱动/线程/服务侧桥接接入层。

### 4.2 核心数据模型（概念层）

1. `entity_key`：唯一实体标识（字符串或句柄映射）。
2. `lease_meta`：`lease_timeout_ms`、`last_heartbeat_ms`、`seq`。
3. `runtime_state`：`ONLINE/OFFLINE`、最近边沿时间、计数器。
4. `event_record`：事件类型、时间戳、原因码、关联 ID、可选标签。

### 4.3 接口草案（待代码冻结）

1. `guard_register(entity_id, lease_timeout_ms, metadata_opt, callback_opt)`
2. `guard_heartbeat(handle, seq_opt, timestamp_opt)`
3. `guard_unregister(handle, reason_code)`
4. `guard_subscribe(filter_opt, event_callback)`
5. `guard_query_snapshot(entity_or_handle)`

约束：

1. 核心主键为 `entity_id`，`metadata` 只用于观测与过滤。
2. 心跳续租只更新租约与状态，不携带业务策略语义。
3. 回调触发仅边沿化，不做周期性重复通知。

### 4.4 执行模型

1. ISR 上下文只允许最小事件提交，不执行回调。
2. 状态推进与业务回调在异步执行上下文完成。
3. 检测与分发解耦，避免慢回调影响超时判定。
4. 所有时间比较统一使用单调时钟口径。

### 4.5 兼容接入策略

1. 对现有驱动（例如电机驱动）采用桥接适配，不改公开 API。
2. 对线程与服务提供统一注册/心跳入口。
3. 旧路径与新守护能力通过功能开关并行，支持渐进迁移。

### 4.6 开源最佳实践映射

| 参考体系 | 设计借鉴 | OM 落地方式 |
| --- | --- | --- |
| systemd Watchdog | 心跳/超时契约与失败动作解耦 | 租约判定在守护核心，恢复动作上交 systems |
| etcd Lease | lease 生命周期与 keepalive 语义 | 注册/续租/撤销作为统一控制面动作 |
| Kubernetes Probes | 检测事实与恢复策略分层 | 守护输出状态事实，业务层决定处置动作 |
| Zephyr Task Watchdog | 多实体监测与统一喂狗接口 | 驱动/线程/服务统一桥接到守护注册中心 |

## 5. 验证方法与结果

### 5.1 计划验证项

1. 合同验证：
   1. 注册/续租/注销行为可判定且返回语义一致。
   2. 离线与恢复均按边沿触发，不重复刷屏。
2. 并发验证：
   1. 回调上下文不在 ISR。
   2. 注册/心跳/注销并发下状态不分裂。
3. 时基验证：
   1. 单调时钟回绕场景判定稳定。
   2. 抖动与丢帧注入下误报率可控。
4. 兼容验证：
   1. 现有驱动公开 API 无破坏性变更。
   2. 关闭守护能力时现有业务行为不变。

### 5.2 当前结果（方案阶段）

1. 文档方案已完成。
2. 代码实现尚未开始。
3. 验证结论将在实现阶段回填。

## 6. 风险与回滚

### 6.1 主要风险

1. 双事实源：桥接层与守护中心状态不一致。
2. 回调阻塞：慢回调拖慢检测与分发。
3. 时基偏差：混用时钟导致误判离线。
4. 层级倒置：实现中引入 `services(core)->drivers` 反向依赖。

### 6.2 风险缓解

1. 强制守护注册中心为唯一在线状态源。
2. 回调异步化并增加背压统计。
3. 统一单调时钟 API 与回绕安全比较。
4. 在评审与 CI 中增加分层依赖检查。

### 6.3 回滚策略

1. 功能开关关闭守护能力，回退到现有驱动本地在线判定路径。
2. 适配层按模块独立启停，支持分批回滚。
3. 保留事件与日志证据，便于快速定位回滚原因。

## 7. 结论与下一步

1. 守护服务实现路径已收敛为：`services` 控制面 + `async` 执行抽象 + 适配层桥接。
2. 对外语义保持对象无关核心模型，核心以 `entity_id + lease` 驱动。
3. 下一步建议：
   1. 在 `services/diagnostics` 冻结头文件契约。
   2. 先落最小 reference 实现，再扩展观测与策略集成。
   3. 逐步接入驱动/线程/服务，并执行分阶段回归验证。
