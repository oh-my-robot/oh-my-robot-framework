# C3_守护服务语义冻结

## 1. 背景与目标

为满足 OM 在驱动、线程、服务等多类模块上的统一在线监测需求，需要在 `services` 层冻结一套与业务对象解耦的守护语义合同。 

本文目标是冻结“机制语义”，为后续实现方案与代码落地提供唯一语义输入。

本文件属于语义冻结文档，不冻结实现级 API 签名、结构体字段布局与平台特定优化细节。

## 2. 范围与非目标

### 2.1 范围

1. 冻结守护实体的统一标识语义。
2. 冻结租约心跳（lease/heartbeat）语义。
3. 冻结在线状态与边沿事件语义。
4. 冻结离线行为语义：上报 + 可注册回调。
5. 冻结与 `services/diagnostics`、`async`、`systems` 的职责边界。

### 2.2 非目标

1. 不冻结具体实现接口、函数参数和内存布局。
2. 不在守护核心内定义自动恢复、自动重启等业务策略。
3. 不绑定具体 RTOS 或具体异步队列后端。
4. 不在本文定义驱动适配细节与板级实现细节。

## 3. 依赖与边界

1. 分层依赖遵循 `oh-my-robot/document/architecture/分层与依赖规范.md`。
2. 守护控制面位于 `lib/services/diagnostics`，仅依赖 `core/osal/sync/async` 抽象能力。
3. 守护执行面依赖 `lib/async` 抽象，不直接依赖具体工作队列实现。
4. `drivers`、线程模块与服务模块通过桥接适配接入守护中心。
5. 业务恢复策略归属 `lib/systems`，不下沉到守护核心。

## 4. 语义冻结条目

### 4.1 实体模型（去类型化核心）

1. 守护核心以 `entity_id` 作为主键，不强制要求 `object_type`。
2. 可选元数据用于观测与过滤（例如 `class/tag`），不参与核心在线判定。
3. 同一 `entity_id` 在同一守护域内必须唯一。

### 4.2 租约与心跳语义

1. 注册动作建立实体租约并进入在线监测域。
2. 心跳动作用于续租，不携带业务策略指令。
3. 注销动作显式回收租约并退出监测域。
4. 租约判定统一基于单调时钟口径。
5. 超时判定结果必须可审计，可追溯到最后有效心跳时间。

### 4.3 在线状态语义

1. 首版状态集合冻结为：`ONLINE`、`OFFLINE`。
2. 合法转换：
   1. `REGISTER -> ONLINE`
   2. `ONLINE -> OFFLINE`（租约超时）
   3. `OFFLINE -> ONLINE`（恢复心跳）
   4. `ONLINE/OFFLINE -> UNREGISTERED`
3. 禁止同一实体在同一时刻出现双状态并存。

### 4.4 事件与回调语义

1. 事件类型冻结为：`REGISTERED`、`ONLINE_EDGE`、`OFFLINE_EDGE`、`UNREGISTERED`。
2. 回调触发采用边沿语义：仅状态变化时触发。
3. 离线动作固定为“上报事件 + 触发可注册回调”。
4. 守护核心不承诺自动恢复语义。
5. 回调执行与检测循环必须解耦，避免慢回调阻塞状态推进。

### 4.5 并发与上下文语义

1. ISR 上下文仅允许最小提交动作，不允许执行业务回调。
2. 业务回调应在异步执行上下文中触发。
3. 共享状态更新必须具备一致的并发保护语义。

### 4.6 现成接口优先决策

1. 时基语义优先复用 OSAL 单调时间能力。
2. 异步提交与执行上下文优先复用 `lib/async` 抽象能力。
3. 不新增与现有语义近似且无差异价值的重复接口。

### 4.7 开源最佳实践对标结论

| 参考体系 | 采纳点 | 不采纳点 | 适配结论 |
| --- | --- | --- | --- |
| systemd Watchdog | 心跳+超时检测、失败动作与检测解耦 | 进程/服务单元绑定模型 | 采纳 lease 心跳思想，不绑定 systemd unit 语义 |
| etcd Lease | `grant/keepalive/revoke` 的租约模型 | 分布式 KV 绑定语义 | 采纳租约状态机，不引入 KV 依赖 |
| Kubernetes Probes | 存活检测与恢复策略分离 | 容器编排内置动作 | 采纳“检测事实与恢复决策分层” |
| Zephyr Task Watchdog / ESP-IDF TWDT | 多任务监测、统一喂狗约束 | 平台专有接口与硬件绑定细节 | 采纳多实体监测思路，不绑定平台私有接口 |

## 5. 验证方法与结果

### 5.1 验证方法

1. 语义一致性审查：检查是否与分层依赖规范冲突。
2. 边界一致性审查：检查是否将业务恢复策略下沉到守护核心。
3. 合同完整性审查：检查实体、租约、状态、事件四类语义是否完整。

### 5.2 当前结果（文档冻结阶段）

1. 语义条目覆盖完整：通过。
2. 分层边界保持清晰：通过。
3. 实现级接口未提前冻结：通过。

## 6. 风险与回滚

### 6.1 主要风险

1. 双事实源风险：驱动局部在线状态与守护中心状态分裂。
2. 时基混用风险：非单调时钟导致误判离线。
3. 回调风暴风险：未做边沿去重或回调隔离。
4. 层级倒置风险：驱动或 OSAL 反向承载守护策略语义。

### 6.2 回滚策略

1. 若实现评审未通过，回退到“驱动局部在线判定 + 业务侧处理”基线。
2. 守护能力采用独立开关控制，默认不影响现有业务链路。
3. 回滚时保留语义文档与证据路径，便于后续再进入实现阶段。

## 7. 结论与下一步

1. 已冻结守护服务首版语义：对象无关模型、租约心跳、双态边沿、离线上报+回调。
2. 下一步进入 `C3_守护服务实现方案.md`，细化组件拆分、接口草案、验证矩阵与迁移路径。
3. 实现阶段需保持与 `lib/async` 抽象层解耦，不绑定具体队列实现。
