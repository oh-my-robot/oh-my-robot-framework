# 文档治理规范（ADR 提纯 + Issue 沙盒 + 版本归档）

## 1. 目的与边界

本规范用于约束开发过程文档的产生、沉淀与归档路径，目标是：

1. 保持 `main` 分支发布资产纯净可审计。
2. 将过程文档、架构决策、公开手册分层管理，避免文档散落与语义漂移。
3. 保证所有关键决策可追溯、可复盘、可归档。

适用范围：

1. 所有开发过程中的需求推演、Bug 分析、阶段计划、实现草案、调试记录。
2. 所有需要长期约束实现行为的架构决策结论。

不适用范围：

1. API 参考手册与就近代码说明文档（保持“代码即文档”原则，就近维护）。

## 2. 动作 1：Issue 沙盒

### 2.1 触发时机

1. 讨论新需求并准备落地草稿。
2. 分析缺陷并形成过程记录。
3. 尚未收敛的架构讨论和阶段性计划。

### 2.2 强制落位

所有过程文档必须放在：

`docs/internal/active/issue_[编号]_[简短slug]/`

示例：

`docs/internal/active/issue_015_motor_pid/`

历史未绑定 Issue 的文档，允许暂存到：

`docs/internal/active/issue_000_[slug]/`

### 2.3 约束

1. 草稿不得直接散落在仓库根目录。
2. 草稿不得写入 `docs/api/`。
3. 草稿不得放入与其无关的代码目录。

## 3. 动作 2：ADR 提纯

### 3.1 触发时机

当复杂机制已形成最终结论并需要长期约束实现时，必须提纯为 ADR。

### 3.2 强制落位与命名

1. 目录：`docs/adr/`
2. 文件命名：`[四位流水号]-[snake_case_slug].md`
3. 示例：`0001-osal_mutex_design.md`

### 3.3 ADR 固定结构

每份 ADR 必须包含且仅包含以下主章节：

1. `背景 (Context)`
2. `考虑过的方案 (Options)`
3. `最终决策 (Decision)`
4. `影响 (Consequences)`

说明：

1. 如有必要，可在 `Decision` 中增加 Mermaid 状态图或时序图。
2. 被提纯后的“决策类文档”应从过程目录或代码目录迁出，避免双事实源。

## 4. 动作 3：版本归档

### 4.1 触发时机

当收到指令：

`【执行版本归档】，当前版本号：vX.X.X`（用户没有提供时你应该停下来询问）

### 4.2 操作要求

1. 扫描 `docs/internal/active/` 下已完成 Issue 目录。
2. 将目录整体迁移到：`docs/internal/archive/[版本号]_features/`
3. 迁移必须使用 `git mv`，保留历史追踪。

### 4.3 命令模板

```bash
git mv docs/internal/active/issue_015_motor_pid docs/internal/archive/v0.3.0_features/
```

## 5. 命名与排版规范

1. 目录与文件使用前缀 + `snake_case`。
2. 文档结构使用清晰标题与有序列表，避免混乱段落堆叠。
3. 路径引用必须使用仓库内真实路径，禁止虚构路径。

## 6. 禁止项

1. 禁止在仓库根目录保存开发过程草稿。
2. 禁止在 `docs/api/` 放置需求推演和讨论记录。
3. 禁止在多个位置维护同一份决策事实。
4. 禁止跳过 ADR 提纯直接把“最终决策”留在 issue 草稿中长期使用。

## 7. 执行清单（Checklist）

每次文档更新前后，必须自检：

1. 过程文档是否全部位于 `docs/internal/active/issue_*`？
2. 已收敛决策是否已经提纯为 `docs/adr/*.md`？
3. 是否存在同一决策在多个文件重复维护？
4. 版本归档是否按指令迁移并使用了 `git mv`？
5. `active/` 是否只保留进行中的任务？
